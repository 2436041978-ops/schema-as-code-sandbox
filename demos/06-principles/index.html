01 声明式（Declarative）
对比：命令式"逐个改文件" vs 声明式"改 1 个 YAML，自动扩散"
❌ 命令式（Imperative）
📄 src/components/Button.tsx
修改 type="danger" → 手动检查
📄 src/panels/AlertCard.tsx
修改 alert_level 枚举 → 人工核对
📄 api/openapi.yaml
添加 x-intent-binding → 口头同步
📄 k8s/policy.rego
更新 OPA 规则 → 容易遗漏
📄 db/migration.sql
添加 COMMENT → 版本对不上
✅ 声明式（Declarative）
YAML intent-schema-compiler/semantic-tokens.yaml
status.critical: { color: #D32F2F, immutable: true }
▼
✅ Compiler 自动分发到 5 层
✅ L1 Token 层：CSS 变量注入
✅ L2 组件层：Props 类型约束
✅ L3 API 层：OpenAPI 扩展
✅ L4 容器层：OPA Policy
✅ L5 DB 层：DDL 注释
02 正交穿透（Orthogonal Penetration）
横向治理网格铺在纵向业务链路上，不替代、不侵入、只注入
Token
组件
API
容器
DB
❌ 侵入式改造
修改 Ant Design 源码
→ 升级冲突，无法维护
重写 React 组件逻辑
→ 业务代码耦合治理逻辑
替换 API Gateway
→ 迁移成本极高
✅ 正交穿透
Ant Design 不动
→ Compiler 注入 Token 注释
React 组件不动
→ HOC 包裹，Props 自动校验
API Gateway 不动
→ 中间件拦截，业务无感知
03 阻断优于修正（Block over Retry）
LLM 输出漂移时：自动重试引入概率漂移，直接拦截保证确定性
❌ 自动重试（Retry）
🤖 LLM 输出：alert_level: "严重"
❌ 第 1 次重试：alert_level: "危急"（更糟）
❌ 第 2 次重试：alert_level: "紧急"（继续漂移）
❌ 第 3 次重试：alert_level: "重大"（映射到 warning！）
💥 最终输出：语义完全偏离，用户看到"重大"却以为是 critical
✅ 直接阻断（Block）
🤖 LLM 输出：alert_level: "严重"
▼
🛡️ Validator 语义推演："严重" 不在枚举 [P0,P1,P2,P3]
🛡️ 命中同义词黑名单：confidence=0.95

🚫 BLOCK

语义推演失败

自动升级人工审核

04 版本锁定（Version Lock）
所有节点显式声明兼容版本，禁止隐式使用 latest
❌ 隐式 latest（危险）
Registry 发布：v1.2.0（收紧同义词规则）
Runtime 加载：schema_version: "latest"
⚠️ 凌晨 2 点自动升级
💥 生产环境：旧规则产物 + 新 YAML 定义 → 不匹配
🔥 结果：拦截逻辑失效，高危操作直通
✅ 显式版本锁定（安全）
Registry 发布：v1.2.0
Runtime 启动校验：
expected_hash: a3f7b2...
actual_hash: a3f7b2...

⚠️ 版本不匹配告警

Runtime 声明 v1.1.0，Registry 最新为 v1.2.0

期望: v1.1.0
实际: v1.2.0

自动降级：进入 escalate 模式，人工确认后升级

✅ 产物哈希一致，规则生效
✅ 版本不匹配时拒绝启动，不静默失败
